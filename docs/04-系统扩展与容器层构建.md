# 04 — 系统扩展与容器层构建

> 本文涵盖 OMV-Extras 安装、系统级插件部署、Docker 环境初始化，以及 GPU 容器直通。
> **前置依赖**：必须先完成 [03-数据存储与局域网共享](./03-数据存储与局域网共享.md) 中的所有步骤（存储架构及基础共享文件夹已就绪）。
> **完成本文后方可进入下一步**：[05-容器与应用服务](./05-容器与应用服务/说明.md)，按需部署各类应用。

## 1. 安装 OMV-Extras

OMV-Extras 是部署 Docker、Compose 及高级插件的统一基础设施。

### 1.1 脚本安装

通过 SSH 执行官方一键安装脚本：

```bash
wget -O - https://github.com/OpenMediaVault-Plugin-Developers/packages/raw/master/install | bash
```

## 2. 部署系统级核心插件

通过 Web UI → `系统` → `插件` 搜索并安装以下插件：

### 2.1 Hosts 网络加速

安装 `openmediavault-hosts` 插件后，在 Web UI 中添加以下地址以加速 GitHub 访问及 TMDB 刮削：

```text
# raw.githubusercontent.com

# api.themoviedb.org

# opensubtitles.org
104.26.6.244 opensubtitles.org
104.26.7.244 opensubtitles.org
172.67.74.14 opensubtitles.org
```

### 2.2 系统备份 (openmediavault-backup)

用于 OMV **仅宿主系统盘**的防御灾备，备份位置建议选外围的 `Cache` 等纯数据域。

> [!WARNING]
> **方式选型警告**：
> - **`dd` 模式**：底层物理扇区级全量克隆，**绝对不支持增量备份**且极度消耗空间。但它是唯一能保证灾后"写入新优盘即刻原样启动"的无脑还原万能药。
> - **`rsync` 模式**：若因存储有限需要高频备份宿主系统，必须改选 `rsync` 模式以触发增量备份。

### 2.3 闪存保护 (openmediavault-flashmemory) — 按需

针对系统盘安装在 U 盘 / TF 卡的场景，建议安装以延长寿命。如果是 SSD/机械硬盘则跳过。

### 2.4 数据盘灾备大策略

| 方案 | 工具 | 适用场景 | 增量支持 |
| :--- | :--- | :--- | :---: |
| **A. 热备** | OMV 原生 `Rsync`（`服务` → `Rsync`） | 机箱内部多块独立硬盘之间的自动化同步 | ✅ 天生支持 |
| **B. 冷备** | `usbbackup` 插件 | 移动硬盘插入 USB 口 → 自动唤醒克隆 → 完毕后自动弹出 | ✅ 天生支持 |

> [!CAUTION]
> **usbbackup 格式化警告**：
> 若备份盘格式化为 exFAT / NTFS（便于跨平台读写），所有 `UID/GID/ACL` 等 Linux 权限标识将在写入瞬间被降维摧毁。
> - ✅ **可用于**：纯家庭相册及影音（`Content` 媒体域）的备份
> - ❌ **严禁用于**：Docker `Software` 核心配置的备份。灾难恢复后容器数据库将大面积爆出权限互锁级死机

## 3. 安装与配置 Compose 插件（核心环境）

Compose 插件是 OMV 管理容器的核心入口，其安装过程会自动触发 Docker 引擎的部署。

### 3.1 启用 Docker 存储库

OMV-Extras 脚本安装完成后，需手动开启 Docker 仓库以获取相关软件包：

1. 导航到 Web UI → **系统** → **omv-extras**。
2. 找到 **Docker repo** 选项并点击 **开启**。
3. 点击 **保存** 并应用更改。

### 3.2 插件安装

前往 Web UI → `系统` → `插件`：
1. 搜索并安装 `openmediavault-compose`。
2. 该插件**必须排在所有常规插件的最后压轴安装**，用于接管全盘的 Docker 引擎。

> [!CAUTION]
> **🚨 Compose 安装可能遭遇 Docker GPG 签名缺失报错**
> 
> 若在 Web UI 安装时出现报错 `W: OpenPGP signature verification failed ... Missing key 7EA0A9C3F273FCD8`，请参考故障排查专栏：👉 **[06-常见问题与故障排除](./06-常见问题与故障排除.md#22-compose-安装报错docker-gpg-签名缺失)** 获取必杀修复指令。

### 3.3 创建 Docker 专用应用账户

在安装完 Compose 插件后，系统已自动创建 `docker` 用户组。此时我们建立一个**无终端登录权限**的隔离账户，用于统一管理容器运行权限：

前往 Web UI → `用户` → `用户`，新建用户：

| 配置项 | 推荐值 | 说明 |
| :--- | :--- | :--- |
| 用户名 | `appuser` | 专用于 Docker 容器的统一身份 |
| 所属组 | `docker` | 在 Web UI 的"组"下拉框中勾选 |

创建完成后，通过 SSH 查询该账户的 UID 与 GID，后续所有容器均需引用此值：

```bash
id appuser
# 输出示例：uid=1001(appuser) gid=1001(appuser) groups=1001(appuser)
```

### 3.4 完善 Docker 数据域权限

既然 `appuser` 已创建，现在需要为其授予对 `03` 章节中建立的顶级共享文件夹的访问权。

前往 Web UI → `存储器` → `共享文件夹` → `权限`：
- 分别选中 `Software` 和 `Content` 文件夹。
- 为 `appuser` 勾选 **Read/Write** 权限并保存。

### 3.5 配置 Compose 工作目录

1. 导航到 Web UI → `服务` → `Compose` → `设置`。
2. 将**工作目录**指向之前建好的 `Software/Compose` 共享文件夹。
3. 保存并应用黄条，Docker 将自动完成拉取安装与接管。

### 3.6 OMV 8 专属防覆盖 — 脱管斩断操作

> [!WARNING]
> 在 OMV 8 环境中，OMV 底层控制引擎 (`omv-salt`) 会在每次页面变动时强行刷新覆盖 `/etc/docker/daemon.json`。

**解绑与固化方案**：
1. 前往 **OMV Web 面板** → `服务` → `Compose` → `设置`。
2. 找到 **"Docker 存储"** 一栏的输入框（默认填有 `/var/lib/docker`）。
3. **将该输入框彻底清空删白**，保存并点击应用黄条。

### 3.7 配置容器镜像加速源

通过 SSH 编辑 `/etc/docker/daemon.json`（确保已执行 3.6 节脱管）：

```json
{
  "data-root": "/var/lib/docker",
  "live-restore": true,
  "log-driver": "json-file",
  "log-opts": {
    "max-file": "3",
    "max-size": "50m"
  },
  "storage-driver": "overlay2",
  "registry-mirrors": [
    "https://hub.rat.dev",
    "https://docker.m.daocloud.io",
    "https://dockerproxy.com",
    "https://mirror.iscas.ac.cn",
    "https://mirror.baidubce.com"
  ]
}
```

**重启 Docker 生效**：
```bash
sudo systemctl daemon-reload && sudo systemctl restart docker
```

## 4. GPU 容器直通配置（按需）

> [!IMPORTANT]
> **定位说明**：此处**绝对不是**安装显卡原生驱动，而是安装让 Docker 引擎能够调用系统 GPU 的驱动插件。详细驱动安装见 [02 章节](./02-基础环境优化.md#4-宿主机-gpu-内核驱动安装按需)。
>
> **强内序依赖**：必须在 Docker 服务完全启动后（§3 完成后）紧接着执行。

参考链接：
- [NVIDIA Container Toolkit 安装指南](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html)
- [Docker GPU 资源约束文档](https://docs.docker.com/engine/containers/resource_constraints/#gpu)

### 4.1 Intel 核显

Intel 核显无需额外 Toolkit，直接在 Compose 中映射设备路径即可：

```yaml
services:
  jellyfin:
    # ... 省略其他配置
    devices:
      - /dev/dri:/dev/dri  # 将宿主机的全部核显设备直通给容器
```

### 4.2 NVIDIA 独显

> 确保已执行 §3.6 的脱管操作，否则配置将被覆盖。

**第一步：引入并签署 NVIDIA 官方软件源**

#### 方案 A：国内加速镜像（推荐）

若遇到 GitHub 连接重置（Connection reset），请执行以下 **中科大（USTC）** 镜像源指令：

```bash
curl -fsSL https://mirrors.ustc.edu.cn/libnvidia-container/gpgkey \
  | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg \
  && curl -s -L https://mirrors.ustc.edu.cn/libnvidia-container/stable/deb/nvidia-container-toolkit.list \
  | sed 's#deb https://nvidia.github.io#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://mirrors.ustc.edu.cn#g' \
  | sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
```

#### 方案 B：官方原源（备选）

```bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey \
  | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg \
  && curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list \
  | sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' \
  | sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
```

**第二步：安装 NVIDIA Container Toolkit**

```bash
sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit
```

**第三步：注册并重启**

```bash
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

**验证安装（可选）**

通过以下指令确认工具链已就绪，且 Docker 已正确识别运行时：

```bash
# 1. 确认工具链已在 PATH 中（即使无显卡虚拟机也能通过）
nvidia-ctk --version

# 2. 确认 Docker 已成功注册 nvidia 运行时
docker info | grep -i nvidia
# 预期输出应包含：Runtimes: ... nvidia ...
```

**第四步：Compose 资源预留写法**

在应用了 Toolkit 后，声明独显**不再使用**已淘汰的 `runtime: nvidia` 标签，而必须采用 Docker 官方推荐的"资源预留"架构，显式告知容器需要调用的硬件资源：

```yaml
services:
  jellyfin:
    image: lscr.io/linuxserver/jellyfin:latest
    # ... 省略常规环境变量及卷映射
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1            # 映射 1 张物理显卡（支持多容器通过时间片并发共享）
              capabilities: [gpu] # 向该容器开放核心 GPU 硬件算力
    # --- 变量透传配置 ---
    # 此处解决了您提到的质疑：将 global.env 中的变量真正注入到容器中
    environment:
      - NVIDIA_VISIBLE_DEVICES=${NVIDIA_VISIBLE_DEVICES}
      - NVIDIA_DRIVER_CAPABILITIES=${NVIDIA_DRIVER_CAPABILITIES}
```
