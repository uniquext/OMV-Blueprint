# 02 — 基础环境优化

> 本文涵盖国内 TUNA 镜像源加速、常见依赖安装、终端中文路径乱码修复、SSD 固态硬盘优化，以及宿主机 GPU 内核驱动（按需）。
> **前置依赖**：必须先完成 [01-系统安装](./01-系统安装.md) 中的所有步骤。

## 1. 配置国内 TUNA 镜像源加速

> [!IMPORTANT]
> 此流程包含**强制内序依赖**：sed 替换底层源 → omv-env 设置 OMV 管理源 → omv-salt 生效 → apt update 验证。任何步骤失败都必须停下排查，不得继续。

参考链接：
- [TUNA 镜像站首页](https://mirrors.tuna.tsinghua.edu.cn/)
- [TUNA OpenMediaVault 帮助](https://mirrors.tuna.tsinghua.edu.cn/help/OpenMediaVault/)
- [TUNA Debian 帮助](https://mirrors.tuna.tsinghua.edu.cn/help/debian/)
- [TUNA Docker-CE 帮助](https://mirrors.tuna.tsinghua.edu.cn/help/docker-ce/)

### 1.1 完整一键流程脚本

通过 SSH 登录宿主机后，依次执行以下命令（建议逐段复制，确认无报错后再执行下一段）：

```bash
# ==========================================
# OMV 8 (Debian Trixie) 完整国内镜像源加速流程
# ==========================================

# ── 第一步：替换 Debian 基础源 (针对 /etc/apt/sources.list)
sudo sed -i 's/http:\/\/httpredir.debian.org/https:\/\/mirrors.tuna.tsinghua.edu.cn/g' /etc/apt/sources.list
sudo sed -i 's/http:\/\/deb.debian.org/https:\/\/mirrors.tuna.tsinghua.edu.cn/g' /etc/apt/sources.list

# ── 第二步：部署 OMV 及扩展包加速环境变量
# 修改 OMV 官方库及备用库
omv-env set OMV_APT_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/OpenMediaVault/public"
omv-env set OMV_APT_ALT_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/OpenMediaVault/packages"

# 修改 OMV 管理的 Debian Security 和 Backports
omv-env set OMV_APT_KERNEL_BACKPORTS_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/debian"
omv-env set OMV_APT_SECURITY_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/debian-security"

# 修改 OMV-Extras, Docker 和 Proxmox 源
omv-env set OMV_EXTRAS_APT_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/OpenMediaVault/openmediavault-plugin-developers"
omv-env set OMV_DOCKER_APT_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/docker-ce/linux/debian"
omv-env set OMV_PROXMOX_APT_REPOSITORY_URL "https://mirrors.tuna.tsinghua.edu.cn/proxmox/debian"

# ── 第三步：令上述所有 OMV 环境变量生效并重新生成 .list 文件
sudo omv-salt stage run all

# ── 第四步：更新本地缓存，验证速度与签名是否完全无误
# (注意：如需锁定内核防止 NVIDIA 驱动失效，请先执行下文 1.2 节)
sudo apt update
sudo apt upgrade -y
```

> [!IMPORTANT]
> **配置生效确认**：上述命令行操作完成后，请务必登录 OMV Web 管理界面，点击顶部黄色提示条中的**应用更改**并勾选**确认**。随后建议重启系统以确保所有底层变更完全生效。

### 1.2 内核版本管理 (防止驱动不兼容)

> [!TIP]
> **适用场景**：当您安装了 NVIDIA 独显驱动或其他深度依赖特定内核版本的软件时，建议锁定当前工作的内核版本。这能防止 `apt upgrade` 自动将内核从 `6.12` 提升到 `6.19+` 甚至更高版本，从而避免因内核接口 (API) 变动导致的驱动编译失败。

#### 1.2.1 锁定内核版本
执行锁定后，系统在执行普通 `upgrade` 时将跳过内核更新：

```bash
# 锁定 Debian 标准内核及其头文件 (Meta-packages)
sudo apt-mark hold linux-image-amd64 linux-headers-amd64
```

#### 1.2.2 查看锁定状态
确认哪些软件包已被“冻结”：

```bash
apt-mark showhold
```

#### 1.2.3 解锁内核版本
当您确认新驱动（或先行路径驱动）已支持高版本内核时，可以随时解锁以进行系统升级：

```bash
# 解锁
sudo apt-mark unhold linux-image-amd64 linux-headers-amd64

# 执行升级
sudo apt update && sudo apt upgrade -y
```

## 3. 终端中文路径乱码修复 (SSH 环境)

当 OMV 仅预装基础英文环境时，通过 Mac 等客户端 SSH 连入，会因远端语言包缺失导致 `ls` 输出中文路径变成八进制转义序列（如 `''$'\346\234...'`）。

### 3.1 标准配置流程

```bash
# 第一步：在配置表中解锁必需的中文和英文 UTF-8 语言包
sudo sed -i 's/^# *\(zh_CN.UTF-8 UTF-8\)/\1/' /etc/locale.gen
sudo sed -i 's/^# *\(en_US.UTF-8 UTF-8\)/\1/' /etc/locale.gen

# 第二步：统一编译生成语言包（耗时约数秒）
sudo locale-gen zh_CN.UTF-8
sudo locale-gen en_US.UTF-8

# 第三步：设定系统默认语言环境
sudo update-locale LANG=en_US.UTF-8

# 第四步 (可选)：为当前账户强制挂载英文+UTF-8 双重环境，防止 PAM 未下发
echo "export LC_ALL=en_US.UTF-8" >> ~/.bashrc
echo "export LANG=en_US.UTF-8" >> ~/.bashrc
```

### 3.2 验证

完成上述命令后，**必须**执行 `exit` 彻底断开当前 SSH 会话，然后重新 `ssh` 登入。此时：
- 所有 locale 相关的终端警告均已消失
- `ls` 查看含中文的文件或目录显示正常汉字

> [!NOTE]
> **物理机直连（显示器 + 键盘）特殊说明**：
> Linux 原生纯文本 TTY 终端不自带中文字体渲染能力。即便语言环境正确，中文仍会显示为菱形乱码（◆◆）或保留转义。如需在物理控制台显示中文，需安装 `fbterm` 或 `zhcon` 等中文控制台环境。但在 SSH 客户端（如 Mac Terminal、iTerm2）中，只要语言环境配置正确即可完美渲染。

## 4. SSD 固态硬盘优化（按需）

> [!NOTE]
> 针对使用 SSD 作为系统盘或数据盘的节点。纯机械硬盘用户可跳过本节。

### 4.1 安装 hdparm 工具

```bash
sudo apt update
sudo apt install hdparm
```

### 4.2 检查 TRIM 支持

```bash
# 方法 1：检查 SSD 是否支持 TRIM（将 sdX 替换为你的 SSD 设备名）
sudo hdparm -I /dev/sdX | grep -i TRIM
# 输出示例：
#   * Data Set Management TRIM supported (limit 8 blocks)
#   * Deterministic read data after TRIM

# 方法 2：使用 lsblk 查看丢弃支持
lsblk -D
# 输出示例：
# DISC-ALN DISC-GRAN DISC-MAX DISC-ZERO NAME
#      512B       4G       4G        0 sda
```

### 4.3 启用定期 TRIM

定期 TRIM 由系统定时任务执行，对性能影响最小，**无需修改 fstab 的 discard 选项**：

```bash
# 检查 fstrim 定时器状态
sudo systemctl status fstrim.timer

# 启用并启动定时器（每周自动执行一次）
sudo systemctl enable fstrim.timer
sudo systemctl start fstrim.timer

# 手动执行一次 TRIM 验证效果
sudo fstrim -v /
# 输出示例：
# /: 123.4 GiB (132456789012 bytes) trimmed
```

### 4.4 fstab 挂载选项优化

**获取系统盘根分区的 UUID**：
```bash
# 方法 1
lsblk -f
# 输出示例：
# NAME        FSTYPE LABEL UUID                                 MOUNTPOINT
# nvme0n1
# └─nvme0n1p1 ext4         a1b2c3d4-e5f6-7890-abcd-ef1234567890 /
#                                                                ↑ 这是根分区

# 方法 2
sudo blkid | grep "/"
```

**编辑 `/etc/fstab`**，找到系统盘根分区配置行，添加优化选项：

```text
# 原配置（示例）
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890  /  ext4  defaults  0  1

# 修改为（添加 noatime,commit=60）
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890  /  ext4  defaults,noatime,commit=60  0  1
```

> [!CAUTION]
> 请将 `a1b2c3d4-e5f6-7890-abcd-ef1234567890` 替换为您通过上方命令获取的**实际 UUID**。错误的 UUID 将导致系统无法启动。

### 4.5 挂载选项说明

| 选项 | 作用 | 说明 |
| :--- | :--- | :--- |
| `noatime` | 禁止更新访问时间 | 大幅减少元数据写入，已包含 `nodiratime` 效果 |
| `commit=60` | 每 60 秒同步一次 | 默认 5 秒，增大可显著减少写入频率 |
| `discard` | 实时 TRIM | ❌ 不推荐，用上方的定期 TRIM 替代 |

### 4.6 立即生效与验证

保存 `/etc/fstab` 后，无需重启即可通过以下命令让配置对系统盘（根分区）直接生效：

```bash
# 第一步：重载系统守护进程以识别 fstab 变更
sudo systemctl daemon-reload

# 第二步：原地重新挂载根分区（对系统盘是安全的）
sudo mount -o remount /

# 第三步：验证挂载参数是否包含 noatime 和 commit=60
mount | grep " / "
```

> [!TIP]
> **什么时候建议重启？**
> 虽然热重载是安全的，但如果您怀疑 `fstab` 语法编写有误，执行 `sudo reboot` 是验证系统能否在引导阶段正常识别并挂载硬盘的最稳妥方式。

## 5. 宿主机 GPU 内核驱动安装（按需）

> [!IMPORTANT]
> **定位说明**：此步骤完全属于裸机 Debian 系统级安装（常需编译内核模块 `dkms`），与 Docker 容器 and 任何上层应用毫无关系。GPU 的**容器直通桥接**（如 NVIDIA Container Toolkit）将在 [04-系统扩展与容器层构建 §4](./04-系统扩展与容器层构建.md#4-gpu-容器直通配置按需) 中配置。

参考链接：
- [NVIDIA 驱动安装指南](https://docs.nvidia.com/datacenter/tesla/driver-installation-guide/index.html)

### 5.1 Intel 核显 (QSV / VAAPI)

Intel 核显驱动通常原生集成在 Linux 内核中，多数情况下**无需额外安装**闭源驱动。

**硬件检测**：
```bash
ls -l /dev/dri
# 若成功，应显示 renderD128, card0 等设备节点
```

> 检测通过即代表 Intel 核显已就绪，后续容器直通仅需在 Compose 中映射 `/dev/dri` 设备即可。
```yaml
    devices:
      - /dev/dri:/dev/dri  # 将宿主机的全部核显设备直通给容器使用
```

### 5.2 NVIDIA 独显 (CUDA / NVENC)

> [!WARNING]
> 此步骤需编译内核模块，耗时较长。请确保系统已完成换源且网络畅通。

#### 5.2.1 前置检查与清理
在执行安装前，请确认您的内核版本和显卡状态。若曾安装失败导致 `apt` 锁死，必须执行清理：
```bash
# 1. 检查当前内核版本
uname -r

# 2. [可选] 若 apt 锁死，请强制清理：
sudo apt purge '^nvidia-.*' -y && sudo apt autoremove -y && sudo apt clean
```

#### 5.2.2 路径选择

根据您的内核版本选择对应的分支。**GTX 1650 用户推荐优先尝试分支 B。**

##### **分支 A：标准路径 (适用于 6.12 / 6.1 等稳定内核)**
使用 Debian 13 (Trixie) 官方仓库驱动，稳定性最高。
```bash
# 安装与运行内核匹配的 Headers (元软件包，随内核自动更新)
sudo apt install linux-headers-amd64 -y

# 安装驱动核心及监控工具 (nvidia-smi)
sudo apt install nvidia-driver firmware-misc-nonfree nvidia-smi -y
```

##### **分支 B：先行路径 (适用于 6.19 及以上极新内核)**
> 参考 https://developer.download.nvidia.com/compute/cuda/repos/debian13/x86_64/
使用 NVIDIA 官方维护的 Trixie 仓库，提供 **595 系列** 高版本驱动，原生支持 6.19 内核。
```bash
# 1. 下载并安装针对 Debian 13 的官方 Keyring
wget https://developer.download.nvidia.com/compute/cuda/repos/debian13/x86_64/cuda-keyring_1.1-1_all.deb
sudo dpkg -i cuda-keyring_1.1-1_all.deb

# 2. 安装官方高版本驱动及内核头文件 (DKMS 编译必需)
sudo apt update
sudo apt install linux-headers-amd64 nvidia-driver nvidia-kernel-dkms nvidia-driver-cuda -y
```

#### 5.2.3 验证与生效
**必须重启一次宿主机** 以挂载新生成的内核模块：
```bash
sudo reboot

# 重启后验证：
nvidia-smi
```
> **验证标准**：若能成功打印出显卡的物理硬件表格（显存容量、温度等），代表底层算力地基已打通。
