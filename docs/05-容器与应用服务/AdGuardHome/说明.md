# AdGuardHome 服务说明

本篇为您提供部署 AdGuardHome（局域网 DNS 服务中心）的使用指导，包括环境初始化保护及冲突解决方案。

## 一、部署架构 (Deployment)
AdGuardHome 主要负责全屋的 DNS 查询和广告拦截过滤。

- **网络架构 (`network_mode: host`)**：作为底层流量的核心网关，如果采用默认通过桥接 NAT 转发，其后台审计只能探测到 `172.x.x.x` 这单一的 Docker 容器网卡来源，失去了追踪定位局域网各个设备查询源头的功能。特批开辟 `host` 直连模式，直接附着于 OMV 宿主机网卡，获取真实客户端 IP。
- **存储逻辑 (Service Local 唯一)**：无任何公共 `AppData` 或媒体 `Resource` 层羁绊，采用极其纯净的自产层：
  - `./config`：存储管理员的图形化页面配置及域名屏蔽自定义列表。
  - `./work`：持久化每天的详细 DNS 查询日志库以及基础运行时缓存数据。

## 二、快速启动 (Quick Start)

> [!IMPORTANT] 
> 采用 `host` 模式极其实用，但它会**暴露给底层操作系统两个极其危险的直接端口冲突陷阱**，请务必完全按照下列引导步骤点击。

1. **拉起容器**
   进入 `Compose/AdGuardHome/`，确认 `AdGuardHome.yml` 配置并一键部署，然后打开浏览器访问其临时向导端口：`http://<您的NAS局域网IP>:3000`。

2. **化解第一个致命陷阱：80 号端口（Web管理监控界面冲突）**
   - 进入第二页向导时，“管理 Web 界面监听端口” **默认值为 80**。
   - **🔴 必须修改**：由于 `host` 模式下直接索要主机的 `80`，这会极其粗暴地跟主系统 OMV 的控制页面发生“内讧冲突”进而安装卡死。
   - **动作**：请老老实实将 Web 端监听端口改为 `8000`（或其他您记录好且空闲的端口，且建议同步将 OMV 的 80 端口迁移至如 8888 留出余量）。

3. **化解第二个致命陷阱：53 号端口（DNS监听网络接口冲突）**
   - 在同一个向导页面，“DNS 服务器监听接口” **默认为所有接口 `(0.0.0.0)`**。
   - **🔴 必须修改**：由于 OMV(Debian系)默认开启的 `systemd-resolved` 进程会抢占底层局域接口 `127.0.0.53` 的 53 插槽，选所有接口会导致它立刻产生 `address already in use` 报错崩溃。
   - **动作**：点击下拉菜单，**精准寻找到写着您这台 NAS 固定局域网 IP（如 `192.168.x.x`） 的那个网卡**（通常带有 `eth0` 或 `enp3s` 前缀）。

## 三、网络安全与底层对比辨析

对于 `systemd-resolved` 与 53 端口宿命般的冲突，您始终面临两次抉择，以下为您进行**【释放底层 53 端口全局统治方案】**和**【当前快速方案】**的优劣势分析备忘：

### A. 当前方案（在向导中精准指定 `192.168.x.x`）
- ✅ **优势**：**安全性极强且省时**。底层管家 OMV 依然保留 `127.0.0.53` 这个内网私密管道给自己更新系统或下载镜像使用，而新房客 AdGuardHome 只需要守死 `192.168.x.x` 这唯一的局域网对外大门网卡口服役，两不相干。这几乎是灾备恢复时效率最高的零容错玩法。
- ❌ **弱势**：在更换主路由或重划网段导致 NAS IP 巨变成 `10.0.x.x` 时，由于网卡 IP 跟它写死的监听 IP 没法对应，容器会在重启时迷失而闪崩。

### B. OMV 系统释放全局 53 方案（关闭 Resolved 服务）
如果您有“全栈管理”的强迫症，非要在向导里选择 **“所有接口 `(0.0.0.0)`”**，则必须对底层 OMV 系统开刀以退让：
```bash
sudo nano /etc/systemd/resolved.conf
# 找到 DNSStubListener=yes 并改为 no，让出 53 给 ADH：
# DNSStubListener=no
sudo systemctl restart systemd-resolved
```
- ✅ **优势**：无论 NAS 未来的物理 IP 怎么更替网段，AdGuardHome 都会坚韧不拔地抓取新网卡重新起效。
- ❌ **弱势**：把 OMV 自己本身能否上网的命脉全部交给了容器。万一 Docker 引擎崩盘或 ADH 未启动，整个 OMV 系统会陷入彻头彻尾的断网（既拉不到 GitHub 的脚本也拿不到外界的最新镜像来实施救援），形成典型的“死锁效应”。（除非给 OMV 加设 `114.114.114.114` 等备用网卡级别的 DNS）。

## 四、详细配置参数表 (Configuration Table)

| 选项 | 值/变量 | 修改建议 | 说明 |
| :--- | :--- | :--- | :--- |
| **基础配置** | | | |
| 容器映像 | `adguard/adguardhome:latest` | 🟢 保持默认 | 使用官方稳定版构建 |
| 网络模式 | `network_mode: host` | 🟢 保持默认 | 为了穿透获取真实用户 IP 必备设定 |
| **环境变量 (Environment)** | | | |
| 时区同步 | `TZ` | 🟢 保持默认 | 指向全局变量 `${TZ}`，因 root 主导暂无视 PUID/PGID |
| **持久层映射 (Service Local)** | | | |
| 规则配置 | `./config` | 🟢 保持默认 | 用户设定的过滤规则黑名单等核心配置 |
| 日志数据 | `./work` | 🟢 保持默认 | 查询日志及核心运行高频读写数据库 |

## 五、日常运维须知 (Operations)

> [!CAUTION]
> **绝对禁止在 ADH 内一键开启 DHCP 服务。**由于您在 `host` 模式的“高屋建瓴”下，开启此服务会与家庭内原有主路由器互抢 DHCP 下发权，这将迅速导致全屋智能设备、电脑、手机网络大面积崩盘及瘫痪。

如果您在安装后需要快速更新过滤节点，建议在上游 DNS 服务器中填入稳定高速的组合：
- `223.5.5.5`（阿里云高连通性）
- `114.114.114.114`（基础备灾容错）
