# 06 — 常见问题与故障排除

> 本文档汇总了在 OMV 部署及使用过程中可能遇到的各类疑难杂症、报错提示及对应修复方案。

---

## 1. 基础环境与网络排错

### 1.1 ⚠️ 日期时间自检与假同步排雷

> [!CAUTION]
> **这一步至关重要！** 即使在 Web UI 中已经填写了 NTP 服务器，底层 `chrony` 的 Slew（平滑修正）保护机制也可能导致系统时间"看似同步，实则卡在昨天"。这将引发后续 GPG 签名校验、APT 源更新等一连串致命报错。

#### 自检方法（SSH 登录宿主机）

**第一步：检查系统时间**

```bash
timedatectl status
```

对比输出中的关键字段：
```text
               Local time: Thu 2026-03-26 00:05:37  ← 实际使用的时间（如果与真实时间不符则必须修复）
                RTC time: Fri 2026-03-27 14:35:05  ← 主板硬件时间（通常准确）
System clock synchronized: yes                     ← 假象！时间虽未纠正但进程已连通
```

> **判断原则**：只要 `Local time` 与真实时间不一致，无论 `synchronized` 显示什么，都必须立即执行修复。

**第二步：进一步核查（可选）**

```bash
chronyc tracking
```

```text
Ref time (UTC)  : Wed Mar 25 15:48:25 2026  ← NTP 参考时间卡死在过去
```

#### 强制跳变修复

不要等待 chrony 自动纠正（大偏差下永远不会成功跳变），直接绕过对时模块，从外网 HTTP 响应头抓取绝对时间强行覆盖：

```bash
# 这条命令需要测试
sudo chronyc makestep || sudo ntpdate ntp.ntsc.ac.cn
# 一键修复：强制对时 + 清理 APT 缓存 + 刷新索引
sudo date -s "$(curl -s --head http://baidu.com | grep ^Date: | sed 's/Date: //g')" \
  && sudo apt clean \
  && sudo apt update
```

> [!TIP]
> 修复后再次执行 `timedatectl status` 确认 `Local time` 已与真实时间一致。后续 NTP 的 Slew 机制将在小偏差范围内正常工作。

## 2. 软件源与签名报错

### 2.1 🚨 GPG 签名截断避坑 (`Not live until` / `Missing key`)

> [!CAUTION]
> 若运行更新脚本或安装软件时遭遇成簇出现的签名时间报错，**切忌盲目自行拼凑寻找缺失证书！**
>
> **根本病灶**：系统底层时间大幅度滞后引发的全局数字证书作废拦截。即使在基础安装中配了 NTP，底层 `chrony` 的 Slew 保护在超过一秒误差时也会坚决阻止时间跳变。
>
> **必杀修复**（SSH 执行）：
> ```bash
> sudo chronyc makestep
> sudo date -s "@$(date +%s)" && sudo apt clean && sudo apt update
> ```
> 待无报错完成拉取后，再次运行官方安装脚本即可秒通。

### 2.2 🚨 Compose 安装报错：Docker GPG 签名缺失

> [!CAUTION]
> 若在 Web UI 安装 `openmediavault-compose` 插件时出现报错 `W: OpenPGP signature verification failed ... Missing key 7EA0A9C3F273FCD8`：
> 
> **原因说明**：Debian Trixie (OMV 8) 启用了更严格的 `sqv` 签名验证机制。OMV-Extras 自动生成的源配置指定了签名文件路径为 `/usr/share/keyrings/docker.gpg`。如果该源文件下载异常（如格式错误或内容为空），将导致软件源验证失败。

**修复步骤**（通过 SSH 在宿主机执行）：

重新下载官方公钥，转换为系统要求的二进制格式并覆盖损坏文件：

```bash
sudo curl -fsSL https://mirrors.tuna.tsinghua.edu.cn/docker-ce/linux/debian/gpg | sudo gpg --dearmor --yes -o /usr/share/keyrings/docker.gpg
sudo apt clean && sudo apt update
```

执行完毕且 `apt update` 无报错返回后，刷新 Web UI 重新应用 Compose 配置即可。

## 3. 数据库备份与恢复 (PostgreSQL & Immich)

### 3.1 🚨 Immich 数据库无损恢复 (还原相册、人脸、元数据)

> [!IMPORTANT]
> Immich 的照片原件虽在 `library` 目录，但所有相册分组、人脸识别标签、收藏状态均存储在 PostgreSQL 数据库中。全新搭建时，必须通过恢复备份来还原这些数据。

#### 前提条件
*   **Immich 容器堆栈已通过 OMV Compose 插件部署并首次启动过**（确保容器实例已创建且底层存储卷已挂载）。
*   `backups/` 目录下存在有效的 `.sql.gz` 数据库备份文件。

#### 恢复步骤（SSH 执行）

1. **停止当前服务**：确保没有写入竞争。
   ```bash
   # 必须显式关联环境文件
   docker compose --env-file ../global.env --env-file Immich.env -f Immich.yml down
   ```

2. **清理新生成的空库**：防止主键冲突。
   ```bash
   # 假设在 Compose/Immich 目录下
   sudo rm -rf ./data/*
   ```

3. **创建容器并单独启动数据库**：
   ```bash
   docker compose --env-file ../global.env --env-file Immich.env -f Immich.yml create
   docker start immich-postgres
   sleep 10  # 等待数据库初始化完毕
   ```

4. **执行“防弹版”恢复指令**：
   此命令包含两个核心修正：修正向量查找路径 (`sed`) 以及开启事务熔断保护。

   ```bash
   # 请将路径替换为实际的备份文件路径
   gunzip --stdout /path/to/backup/immich-db-backup.sql.gz \
     | sed "s/SELECT pg_catalog.set_config('search_path', '', false);/SELECT pg_catalog.set_config('search_path', 'public, pg_catalog', true);/g" \
     | docker exec -i immich-postgres psql --dbname=<DB_DATABASE_NAME>  --username=<DB_USERNAME> --single-transaction --set ON_ERROR_STOP=on
   ```

   > [!TIP]
   > **故障排查：如果报错 `invalid command \restrict` 或 `\unrestrict`**
   > 这是由于部分备份工具（或旧版脚本）在 SQL 文件首尾注入了非标准的封包标记。若遇到此报错，请在管道中加入 `grep` 过滤逻辑：
   > ```bash
   > gunzip --stdout /path/to/backup/immich-db-backup.sql.gz \
   >   | grep -v '^\\restrict' | grep -v '^\\unrestrict' \
   >   | sed "s/SELECT pg_catalog.set_config('search_path', '', false);/SELECT pg_catalog.set_config('search_path', 'public, pg_catalog', true);/g" \
   >   | docker exec -i immich-postgres psql --dbname=<DB_DATABASE_NAME>  --username=<DB_USERNAME> --single-transaction --set ON_ERROR_STOP=on
   > ```

5. **全量拉起服务**：
   ```bash
   docker compose --env-file ../global.env --env-file Immich.env -f Immich.yml up -d
   ```

#### 关键参数解析
- **`grep -v '^\\restrict'`**：剔除部分备份脚本在首尾添加的非 SQL 标准干扰指令。
- **`sed ... 'public, pg_catalog'`**：强制装载 `public` 架构，解决 Immich 机器学习向量插件 (`pgvecto-rs`) 恢复时找不到路径的问题。
- **`--single-transaction`**：确保恢复具有原子性，要么全成，要么全败回滚，不产生残留脏数据。

## 4. 性能与告警优化 (Performance & Warnings)

### 4.1 🔧 Redis 内存超售警告 (Memory overcommit)

> [!WARNING]
> **日志报错**：`WARNING Memory overcommit must be enabled!`
> 
> **影响说明**：虽然这不影响目前服务启动，但对于依赖 Redis 作为缓存和队列的任务（如 Immich 的后台处理和转码）而言，未开启 `overcommit` 可能在极端负载下导致持久化保存失败。为了性能和稳定，建议在 OMV 宿主机级别修复。

#### 修复步骤（SSH 登录宿主机执行）

1. **立即生效**：
   ```bash
   sudo sysctl -w vm.overcommit_memory=1
   ```

2. **持久化保存**（确保重启不失效）：
   ```bash
   echo "vm.overcommit_memory = 1" | sudo tee -a /etc/sysctl.conf
   ```

#### 参数原理解析

Linux 内核通过 `vm.overcommit_memory` 参数控制进程的内存分配策略：
- **`0` (默认值/启发式)**：内核会估算可用内存，如果进程申请的内存显著大于系统剩余可用量，会直接拒绝分配。
- **`1` (始终超售)**：内核假定系统总有足够的内存，直接批准所有分配请求。这基于计算机科学中的常见事实：进程通常会申请远多于它实际使用的内存。
- **`2` (严格不超售)**：总分配量绝对不允许超过 Swap 空间加上设定比例的物理内存，超出即报错。

**为什么 Redis 推荐设置为 `1`？**
Redis 在执行后台持久化（BGSAVE）时，会使用操作系统的 `fork` 机制创建一个子进程。借助写时复制（Copy-on-Write）技术，子进程最初并不立刻占用全量内存。但在默认的 `0` 模式下，如果系统剩余空闲内存小于 Redis 当前占用的总量，内核评估机制会错误地认为“如果子进程写满内存系统崩溃的风险太高”从而无情地拒绝 `fork` 请求。这将导致 Redis 后台队列无法落盘甚至程序崩溃。将其设置为 `1` 可确保子进程必定创建成功，保障高负载下的 I/O 安全性。

## 5. 权限与存储排错 (Permissions & Storage)

### 5.1 🔐 ACL 权限继承失效与 "Permission Denied" 修复

> [!CAUTION]
> **场景说明**：在 OMV 的特定共享文件夹内（如 `Compose`），服务或容器自动创建了新的文件或文件夹，但随后其他经过授权的用户（如 `uniquext`）或服务却无法执行删除/修改操作。通过 `getfacl` 检查发现 `effective:r--` 等掩码 (Mask) 被自动收紧。
>
> **核心原因**：Linux 默认新建内容不仅不会自动携带 ACL，还会受 `umask` 限制。只给目录赋权是不够的，必须将"默认继承"（Default ACL）与"掩码拉升"（Mask Reset）结合，才能彻底打通。

#### 修复与锁定流程（SSH 登录宿主机执行）

请严格依次执行以下三步。请将 `<username>` 替换为您想要保护权限的用户名（如 `uniquext`），将 `<target_dir>` 替换为真实的物理挂载地址（如 `/srv/dev-.../Compose`）。

```bash
# 1. 全局覆写：强制赋予该用户在当前整个目录树上所有的 rwx 权限
sudo setfacl -R -m u:<username>:rwx <target_dir>

# 2. 建立继承：设置"默认权限"，让该目录未来产生的所有新建文件/文件夹，出生即带上该用户的 rwx（核心防断层）
sudo setfacl -R -d -m u:<username>:rwx <target_dir>

# 3. 破除屏蔽：统一将最大权限掩码 (Mask) 拉高到 rwx，解除"权限明明给了但是提示 effective:--- "的隐蔽拦截
sudo setfacl -R -m m:rwx <target_dir>
```

> [!TIP]
> **核心逻辑**：
> 第一步是解决"历史遗留"；
> 第二步 (`-d`) 是解决"未来新生"；
> 第三步 (`m:rwx`) 是解决"木桶效应短板"（Linux ACL 会被传统的最高权限遮罩限制，必须强行拉开）。

## 6. Immich 异常排查

### 6.1 🚨 在物理硬盘上直接删除了照片，Immich 里留下了点不开的"幽灵缩略图"？

- **现象**：由于没有走 Web UI，数据库依然认为老照片还在，网页端残留点击后报错的"灰色幽灵缩略图"。
- **原因**：在宿主机上手动删除了图片文件，数据库无法感知。

#### 解决方案

使用预置的自动化清理脚本进行深度清理：

**脚本路径**：`AppData/scripts/immich_cleanup_ghosts.sh`

**执行方法**：

```bash
bash AppData/scripts/immich_cleanup_ghosts.sh
```

**脚本功能**：
1. 自动读取数据库全量资产记录
2. 在 server 容器内校验物理文件存在性
3. 列出所有"失联"的幽灵资产
4. 确认后从数据库中彻底删除

> [!WARNING]
> **核心安全纪律**：**严禁在宿主机直接操作照片文件**。请所有的增删改操作始终通过 Immich 网页版进行。
