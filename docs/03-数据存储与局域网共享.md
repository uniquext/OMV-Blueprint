# 03 — 数据存储与局域网共享

> 本文涵盖存储插件安装、LVM 存储架构搭建、共享文件夹与权限体系，以及 SMB 网络协议配置。
> **前置依赖**：必须先完成 [02-基础环境优化](./02-基础环境优化.md) 中的所有步骤。完成本文后方可部署 Docker 及应用层。

---

## 1. 安装存储关联插件与硬件监控

> [!IMPORTANT]
> 如果您的硬盘计划使用 LVM 逻辑卷架构，**必须**先安装 `openmediavault-lvm` 插件！后续所有 Docker 应用的数据持久化均强依赖本阶段构建的共享文件夹。

### 1.1 存储类插件

进入 Web UI → `系统` → `插件`，搜索并安装：

| 插件 | 功能 | 备注 |
| :--- | :--- | :--- |
| `openmediavault-lvm` | 逻辑卷管理基座 | **必须**在配置存储前提前安装 |

### 1.2 S.M.A.R.T. 硬盘监控

前往 `存储器` → `S.M.A.R.T.` → `设置`：

| 配置项 | 推荐值 | 说明 |
| :--- | :--- | :--- |
| 温度监控 | ✅ 开启 | 实时监测硬盘温度与健康度 |
| 检查间隔 | `3600` 秒 | 每小时执行一次自检扫描 |
| 电源模式 | 待机 | 待机和睡眠模式下不检查设备。此模式下磁盘一般不旋转 |
| 温度监测 | 最大 45°C | 实时监测硬盘温度与健康度 |

## 2. 存储介质初始化与挂载

> [!TIP]
> **构建策略**：强烈建议遵循"**逻辑卷 — 文件系统一一对应**"的架构设计。即使是单块硬盘，也推荐建立 LVM，这将为后期的无缝**在线扩容**及缩容提供极大便利。

### 2.1 标准挂载工作流

在 Web UI 中**严格按以下顺序**执行，每一步保存并应用黄条后再进入下一步：

```mermaid
flowchart LR
    A["① 擦除设备"] --> B["② 物理卷 PV"]
    B --> C["③ 卷组 VG"]
    C --> D["④ 逻辑卷 LV"]
    D --> E["⑤ 文件系统 & 挂载"]
```

| 序号 | Web UI 路径 | 操作 |
| :---: | :--- | :--- |
| ① | `存储器` → `磁盘` → `擦除` | 选中目标磁盘，执行擦除（将清空全部数据） |
| ② | `存储器` → `LVM` → `物理卷` | 选择刚擦除的设备，创建物理卷 |
| ③ | `存储器` → `LVM` → `卷组` | 将物理卷加入卷组（可多块物理卷合并为一个卷组） |
| ④ | `存储器` → `LVM` → `逻辑卷` | 从卷组中划分所需容量的逻辑卷 |
| ⑤ | `存储器` → `文件系统` → `创建` → `挂载` | 为逻辑卷创建文件系统并执行挂载 |

### 2.2 文件系统创建注意事项

| 要点 | 说明 |
| :--- | :--- |
| **格式推荐** | 通常推荐 `BTRFS`，支持快照与在线扩容 |
| **BTRFS Single 警告** | 若选择 `BTRFS Single` 模式，**切忌同时勾选多块磁盘**，以免形成非预期的跨盘阵列 |
| **标签标定** | 创建文件系统时，**务必写入标签**（如 `Software` 或 `Content`），极大提升共享文件夹的可读性与易维护性 |

## 3. 共享文件夹布局（核心基座）

### 3.1 数据分层隔离（两大顶级共享文件夹）

基于业务属性，为新建的底层挂载介质创建两大顶级共享文件夹：

```text
/srv/
├── dev-disk-by-uuid-xxx[Software]     # 系统数据域
│   ├── AppData/       # 公共资源层（多服务共享的静态资源）
│   └── Compose/       # Docker Compose 文件
└── dev-disk-by-uuid-xxx[Content]      # 内容资源域
    ├── Media/         # 媒体库（电影、剧集、音乐）
    ├── DCIM/          # 相机照片
    ├── Cloud/         # 云盘数据
    ├── Download/      # 下载目录
    ├── Cache/         # 缓存目录
    └── ...            # 其他共享文件夹
```

| 域 | 变量 | 用途 |
| :--- | :--- | :--- |
| **Software** | `${COMPOSE_DATA_PATH}` 指向 `AppData` | 容器及系统运行核心数据 |
| **Content** | `${RESOURCE}` | 私有云及多媒体娱乐资源 |

### 3.2 权限授予策略

> [!WARNING]
> 必须**明确区分**宏观网络鉴权（Privileges）与物理级锁印（ACL），初期**严禁**无脑大面积使用 ACL。

#### 常规授权 (Privileges) — 首选方案

前往 `共享文件夹` → `权限`：
- 给新建的两个顶级域（`Software` / `Content`）分配普通家庭用户账户的 **Read/Write** 权限。
- **关于 Docker 权限**：针对后续容器运行所需的 `appuser` 身份及其专用权限，将在 [04-系统扩展与容器层构建](./04-系统扩展与容器层构建.md) 安装 Docker 引擎后再统一配置。

#### 底层授权 (ACL 靶向排错) — 仅限特殊情况

> [!CAUTION]
> **日常态极不推荐盲目大面积使用！**

仅当部署诸如 MariaDB 或 PostgreSQL 等对硬盘归属权存在极致校验的容器且抛出权限报错时，才需要使用 ACL 强压权限，详见后续章节。

## 4. 配置 SMB 网络共享协议

在完成上方存储与权限架构后，启用 SMB 协议供内网终端无缝接入。

前往 Web UI → `服务` → `SMB/CIFS` → `共享`，为相关的共享文件夹启用 SMB 服务。

### 高阶网络选项调校

| 选项 | 推荐状态 | 说明 |
| :--- | :---: | :--- |
| **可浏览 (Browsable)** | ✅ 开启 | 在局域网广播中开放挂载名，实现 Win/Mac 网络邻居可视化双击挂载 |
| **继承权限 (Inherit Permissions)** | ✅ 开启 | 保证家中任意有权限用户上传的子文件夹和文件，其他被授权用户均可直接读改，避免"只要不是我传的就碰不了"的权限孤岛 |
| **继承 ACLs (Inherit ACLs)** | ❌ 保持默认关闭 | 维持原封不动默认态。避免让复杂且危险的 POSIX ACL 随远程拖拽拷贝操作被失控级联污染 |
